# 简历与面试指南

## 1. 先说结论

这个项目完全可以写进简历，而且适合放在“嵌入式 / Linux 系统 / 软硬件协同 / 边缘智能终端”方向的项目经历中。

它的价值不在于页面是否漂亮，而在于它覆盖了完整链路：

- 设备树
- Linux 驱动
- GPIO/I2C
- 字符设备协议
- V4L2 图像采集
- 多线程服务
- HTTP 接口
- 前后端联动

如果你能把实现思路讲顺，这会比普通 CRUD 项目更有辨识度。

## 2. 简历里该怎么定位这个项目

推荐定位方式：

- 嵌入式智能监控终端
- 基于 ROCK 3A 的多传感器联动系统
- Linux 驱动与用户态协同的边缘监控系统

不太建议定位成：

- AI 监控平台
- 复杂视觉识别系统
- 工业级告警平台

原因很简单：当前代码里并没有做模型推理、复杂视觉分析、权限系统、云端平台或工业级可靠性体系。简历表述要强，但不能脱离实现。

## 3. 一段适合简历项目标题下方的简介

## 4. 简历描述示例

## 4.1 三条版

- 基于 Linux 平台驱动与设备树 overlay 实现自定义 `sensor_hub` 字符设备，统一接入 PIR 人体红外与 SHT20 温湿度传感器，向用户态提供 `ioctl/read/poll` 访问接口。
- 基于 V4L2、TurboJPEG 与多线程机制实现双模式图像链路：`monitor` 模式下提供 `1280x720@30fps` MJPEG 实时预览，`trigger` 模式下在 PIR 触发后完成 `1920x1080` 单帧抓拍与日志留存。
- 自研轻量级 HTTP 服务与 Web 控制台，支持模式切换、实时状态查询、事件日志查看和最新抓拍图像展示，完成从底层驱动到前端可视化的端到端闭环。

## 4.2 四条版

- 设计并实现基于 ROCK 3A 的双模式监控终端，将实时预览与事件抓拍拆分为 `monitor` / `trigger` 两种工作模式，平衡实时性与图像质量。
- 编写 Linux `sensor_hub` 驱动，统一管理 GPIO 中断型 PIR 与 I2C 型 SHT20，抽象出统一事件流和状态快照模型，支持配置下发与强制刷新。
- 基于 V4L2 多平面采集接口完成摄像头接入，构建“NV12 采集 -> JPEG 编码 -> MJPEG 推流”链路，并实现模式切换时的摄像头重配与失败回滚。
- 实现用户态守护进程、日志记录和 Web 控制台，支持状态展示、模式控制、日志查看和抓拍结果回显。

## 4.3 一句话精简版

> 独立完成基于 ROCK 3A 的多传感器联动监控终端，实现 Linux 驱动、V4L2 图像采集、事件触发抓拍、MJPEG 预览与 Web 控制台。

## 5. 你真正可以强调的技术亮点

下面这些点都能从当前代码里找到依据，适合在面试中重点展开：

### 5.1 统一驱动协议设计

不是直接让用户态分别访问 GPIO 和 I2C，而是抽象出了统一的 `/dev/sensor_hub`。

这可以体现你对“模块边界”和“可扩展性”的理解。

### 5.2 双模式设计

不是用一套链路硬扛所有目标，而是把：

- 实时预览
- 事件抓拍

拆成两套不同参数和不同处理路径。

这体现的是工程取舍能力。

### 5.3 事件流 + 快照双模型

驱动既有事件队列，也有全量状态快照：

- 事件队列适合 PIR 触发
- 快照适合 monitor 状态展示

这比只做简单轮询更完整。

### 5.4 摄像头链路不是“调用现成工具显示视频”

当前实现直接操作 V4L2、MMAP buffer、队列出入队、YUV 到 JPEG 转换和 MJPEG HTTP 输出，这些都是比较有含金量的底层工作。

### 5.5 模式切换考虑了失败回滚

很多 demo 项目做到“能切换”就结束了，但这里主线程在切换失败时会尝试恢复旧模式。这是可以拿来讲工程稳定性的。

## 6. 面试时建议怎么讲

一个比较稳的讲法是“四段式”：

### 第一段：项目目标

先说明要做一个嵌入式监控终端，既要能实时预览，也要能在 PIR 触发时抓拍并记录环境数据。

### 第二段：架构拆分

再说系统分成三层：

- 内核驱动统一接传感器
- 用户态守护进程负责编排和服务
- Web 控制台做展示和控制

### 第三段：你最重要的设计选择

重点讲两个：

- 为什么做统一的 `sensor_hub`
- 为什么拆成 `monitor` 和 `trigger`

### 第四段：落地效果

最后讲最终能做到：

- 网页实时看画面
- 实时查看温湿度/PIR 状态
- 触发后保存 JPG
- 自动记录事件日志

这样讲层次最清楚。

## 7. 可能被追问的问题

下面这些问题非常可能在面试里出现。

## 7.1 为什么不让用户态直接访问 GPIO 和 I2C

推荐回答方向：

- 用户态接口会变得分散
- 扩展新传感器时耦合度更高
- 统一协议后，上层只认 `sensor id + snapshot + event`
- 更适合长期演进

## 7.2 为什么 monitor 和 trigger 不共用一套摄像头参数

推荐回答方向：

- 实时预览更看重流畅性与处理开销
- 触发抓拍更看重单帧质量
- 分别使用 `1280x720@30fps` 和 `1920x1080` 是面向目标的配置优化

## 7.3 为什么驱动协议里不用 float

推荐回答方向：

- 内核与用户态共享结构体更适合使用定点整数
- ABI 更稳定
- 避免浮点带来的兼容与表达问题
- 传感器值转换放在显示层完成即可

## 7.4 为什么 MJPEG 采用内存缓存而不是写文件

推荐回答方向：

- 减少磁盘 I/O
- 实时性更好
- 浏览器直接兼容 `<img src="/stream.mjpg">`
- 适合小型嵌入式场景快速落地

## 7.5 这个项目里最有挑战的部分是什么

推荐回答方向：

- 统一抽象 PIR 与 SHT20 的驱动协议
- 处理模式切换时摄像头的关闭、重开和线程同步
- 把传感器触发、图像采集、日志记录、网页展示串成一个完整闭环

## 9. 关于“vibe coding”这件事，怎么诚实但不减分

如果面试官追问开发方式，最稳妥的表达不是回避，而是诚实说明：

> 项目开发过程中使用了 AI 辅助编码来加速样板代码与接口搭建，但系统方案、模块边界、驱动协议、联调验证和问题排查是我自己梳理并吃透的。我后来专门把架构、链路和关键实现重新整理过，确保自己能独立解释和继续维护这套系统。

这种说法通常比硬装“全手写”更可信。

## 10. 你至少要吃透的 10 个知识点

如果你准备把这个项目写进简历，建议你把下面 10 个点真正学明白：

1. 平台驱动和设备树 overlay 是怎么配合的
2. `miscdevice` 与普通字符设备的关系
3. `ioctl/read/poll` 各自适合什么场景
4. GPIO 中断为什么适合 PIR
5. I2C 设备为什么适合用周期刷新
6. `snapshot` 与 `event queue` 为什么要同时存在
7. V4L2 的 `REQBUFS/QBUF/DQBUF/STREAMON` 流程
8. NV12、I420、JPEG 之间是什么关系
9. MJPEG 推流为什么能被浏览器直接显示
10. 模式切换时为什么要让主线程统一编排

把这 10 个点吃透后，这个项目就不只是“可以写在简历上”，而是“你真的能守住面试追问”。
