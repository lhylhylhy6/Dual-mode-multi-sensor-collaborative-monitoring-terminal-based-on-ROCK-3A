# 项目总览

## 1. 项目定位

这个项目的完整名称是“基于 ROCK 3A 的双模式多传感器联动监控终端”。它不是单一的传感器采集 demo，也不是单一的网页前端，而是一个跨越硬件、内核、用户态服务、Web 控制台的端到端系统。

从能力边界来看，它已经具备一个小型嵌入式监控终端的核心闭环：

- 能采集环境与触发类传感器数据
- 能在事件发生时抓拍图像
- 能通过网页对系统进行查看与控制
- 能在“实时预览”和“事件抓拍”两类目标之间切换

## 2. 解决的问题

很多嵌入式项目要么只做“持续采集”，要么只做“触发报警”。这个项目把两种目标分开处理：

- `monitor` 模式：强调实时查看，适合日常巡检和远程观察
- `trigger` 模式：强调事件驱动，适合 PIR 触发后的高质量抓拍和日志留痕

这种设计的核心价值是把“实时性”和“抓拍质量”拆开，避免一个模式同时承担所有目标。

## 3. 硬件组成

当前代码中已经接入的硬件如下：

- 主控板：Radxa ROCK 3A
- 摄像头：Raspberry Pi Camera Module v2
- 人体红外传感器：HC-SR501 PIR
- 温湿度传感器：SHT20，挂在 `i2c-2`，地址 `0x40`
- 无源蜂鸣器：接在 `PWM9_M0 / PIN18`，通过 `sensor_hub` 作为 PWM 输出端点统一管理

代码中还保留了后续继续扩展更多传感器的空间，例如：

- 光照
- 烟雾
- 门磁
- 自定义传感器 ID

## 4. 软件分层

项目可以分成三层：

| 层级 | 位置 | 主要职责 | 关键技术 |
| --- | --- | --- | --- |
| 内核态 | `kernel/sensor_hub` | 统一管理输入/输出端点，当前接入 PIR、SHT20 与 PWM 蜂鸣器 | Linux 平台驱动、字符设备、GPIO IRQ、I2C、PWM、`ioctl/read/poll` |
| 用户态 | `user/daemon` | 读取传感器、管理摄像头、切换模式、抓拍、写日志、提供 HTTP 服务 | C、V4L2、pthread、socket、TurboJPEG、`ffmpeg` |
| 前端 | `user/web/index.html` | 查看状态、切换模式、显示实时预览和抓拍结果 | HTML、CSS、原生 JavaScript |

## 5. 核心设计思想

### 5.1 统一端点入口

项目没有让用户态分别直接操作 GPIO、I2C、PWM 等硬件接口，而是通过一个统一字符设备 `/dev/sensor_hub` 提供访问能力。这样用户态只需要学习一套协议：

- `read/poll`：读事件
- `ioctl(GET_SNAPSHOT)`：读当前快照
- `ioctl(FORCE_REFRESH)`：主动刷新某个传感器
- `ioctl(GET/SET_SENSOR_CFG)`：读写配置
- `ioctl(RUN_ACTION)`：触发输出动作，例如蜂鸣器 `alert/pulse/stop`

这对于后续增加新输入或输出端点都很重要，因为用户态接口可以保持稳定。

### 5.2 双模式分工

`monitor` 与 `trigger` 的目标并不一样：

- `monitor` 模式使用 `1280x720@30fps`，重点是持续预览
- `trigger` 模式使用 `1920x1080` 单帧抓拍，重点是事件发生时保留清晰图像

这也是这个项目最值得写进简历的设计点之一，因为它体现了“根据业务目标配置采集链路”的工程取舍。

### 5.3 事件流和状态快照并存

项目同时维护两种状态/控制方式：

- 事件流：适合 `trigger` 模式，靠 PIR 中断触发
- 状态快照：适合 `monitor` 模式，周期读取全部传感器状态
- 动作接口：适合蜂鸣器这类输出端点，由用户态显式触发

这比只做轮询更合理，也比只做事件更完整，也让驱动从“统一输入”自然扩展到了“统一输入/输出”。

## 6. 两种模式的端到端链路

### 6.1 `monitor` 模式

`monitor` 模式的目标是“网页持续看到最新画面，同时看到当前传感器状态”。

链路如下：

```text
/dev/video0
  -> V4L2 连续采集 NV12 帧
  -> monitor 线程把 NV12 转成 I420
  -> TurboJPEG 压缩成 JPEG
  -> 发布到 HTTP 服务内部缓存
  -> 浏览器通过 /stream.mjpg 持续拉取

/dev/sensor_hub
  -> 用户态每 1 秒获取一次 snapshot
  -> 更新 app_state
  -> /api/status 返回当前模式、PIR、温湿度、最近状态
```

这个模式下系统更像一个轻量级实时监控台。

### 6.2 `trigger` 模式

`trigger` 模式的目标是“等事件来了再做一次完整抓拍”。

链路如下：

```text
PIR GPIO 中断
  -> sensor_hub 生成 PIR 事件
  -> 用户态 poll/read 收到事件
  -> RUN_ACTION(BUZZER, ALERT)
  -> FORCE_REFRESH SHT20
  -> GET_SNAPSHOT 读取事件时刻的传感器快照
  -> /dev/video0 抓一帧 NV12 原始图像
  -> 调用 ffmpeg 转成 JPG
  -> 写入 output/snap_*.jpg
  -> 追加写入 output/events.log
  -> Web 页面可查看最新抓拍结果
```

同时，`trigger` 模式下用户态也会每秒刷新一次 snapshot，这样网页上的 PIR 电平和蜂鸣器状态能在事件结束后自动回落，不会长期停在旧值。

这个模式下系统更像一个带提醒能力的事件录像/抓拍终端。

## 7. 当前已经实现到什么程度

从代码来看，这个项目不是“只有框架”，而是已经完成了比较完整的最小可用系统：

- 自定义内核驱动可工作
- PIR 已通过 GPIO 中断进入统一驱动
- SHT20 已通过 I2C 接入统一驱动
- 无源蜂鸣器已通过 PWM9_M0 接入统一驱动
- 用户态可通过共享头文件访问驱动协议
- 摄像头采集已打通
- monitor 模式的 MJPEG 预览已实现
- trigger 模式的蜂鸣器提醒、抓拍和日志已实现
- 网页控制台已支持模式切换、状态查看、蜂鸣器状态查看、日志查看和图像查看

## 8. 项目产出物

系统运行后，会产出以下结果：

- `output/snap_*.jpg`：触发模式下抓拍得到的照片
- `output/events.log`：事件日志
- `GET /api/status`：实时状态接口
- `GET /stream.mjpg`：实时视频流

示例日志如下：

```text
2026-04-15 09:44:18 MODE=trigger PIR=1 BUZZER=1 TEMP=23.81C HUMI=56.16%RH IMG=output/snap_20260415_094416_1920x1080_nv12.jpg
```
